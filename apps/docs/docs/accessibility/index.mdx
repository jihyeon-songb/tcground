# 접근성 사례 (Do / Don't)

접근성은 문서에만 적는 체크리스트가 아니라 컴포넌트 API의 일부입니다.
이 섹션은 **같은 UI를 잘못 만든 예(❌)와 올바르게 만든 예(✅)로 나란히 비교**하며,
무엇이 왜 문제인지와 `@tcground/headless`가 그것을 어떻게 해결하는지를 보여줍니다.

겉보기에는 두 버전이 거의 똑같지만, 키보드와 보조 기기 앞에서는 전혀 다르게 동작합니다.

## 공통 원칙

- 키보드만으로 열기, 이동, 선택, 닫기가 가능해야 합니다.
- 시각적 focus ring은 제거하지 않고 테마 토큰으로 관리합니다.
- 상태는 `aria-expanded`, `aria-selected`, `aria-checked`, `aria-modal`처럼 보조 기술이 이해할 수 있는 속성으로 노출합니다.
- uncontrolled와 controlled 상태를 모두 지원해 앱 요구사항에 맞게 상태를 소유할 수 있게 합니다.

## 왜 직접 만들면 깨지기 쉬운가

`<div onClick>` 으로도 "눌리는 것처럼 보이는" UI는 쉽게 만들 수 있습니다. 하지만 그렇게 만든 컴포넌트는 보통 다음을 빠뜨립니다.

- **역할(role)**: 보조 기기는 `<div>` 를 버튼/탭/라디오로 인식하지 못합니다.
- **키보드 조작**: 클릭 핸들러만 있으면 마우스 사용자만 쓸 수 있습니다.
- **focus 관리**: 모달을 열어도 focus가 따라가지 않거나, 닫은 뒤 원래 위치로 돌아오지 않습니다.
- **상태 노출**: 선택/펼침 상태가 시각적으로만 표현되어 스크린리더가 읽지 못합니다.

`@tcground/headless` primitive는 이 동작들을 기본으로 구현해 두었고, `@tcground/ui` 는 그 위에 스타일만 입힙니다.

## 직접 확인하는 법

각 사례 페이지의 ❌/✅ 데모를 아래 방법으로 비교해 보세요.

- **키보드만 사용**: 마우스를 치우고 `Tab`, 방향키, `Enter`/`Space`, `Esc` 만으로 조작해 봅니다.
- **focus 추적**: 지금 어디에 focus가 있는지 focus ring으로 확인합니다. 모달을 닫은 뒤 focus가 트리거로 돌아오는지 봅니다.
- **접근성 트리**: 브라우저 DevTools의 Accessibility 패널에서 각 요소의 role과 `aria-*` 속성을 확인합니다.
- **스크린리더**(선택): VoiceOver(macOS) 또는 NVDA(Windows)로 요소가 어떻게 읽히는지 들어 봅니다.
